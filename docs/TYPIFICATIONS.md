# vCenter Provisioner - Typifications Documentation

## Overview

The Typification System provides a structured naming convention for virtual machines (VMs) in vCenter. This system ensures consistency, traceability, and automation of VM naming across the infrastructure.

## Naming Convention

### Structure

VMs follow a fixed naming pattern:

```
{prefijo1}-{prefijo2}-{manual_value}-{sequence}
```

### Components

1. **Prefijo 1**: First prefix (e.g., `SRV`, `PRD`, `TST`)
   - Alphanumeric only (letters and numbers)
   - Maximum 50 characters
   - Defined when creating a typification

2. **Prefijo 2**: Second prefix (e.g., `WEB`, `DB`, `APP`)
   - Alphanumeric only (letters and numbers)
   - Maximum 50 characters
   - Defined when creating a typification

3. **Manual Value**: User-provided value at VM creation time
   - Alphanumeric only (letters and numbers)
   - No length restriction
   - Provided when creating a VM
   - Represents a specific identifier (e.g., project code, owner, environment)

4. **Sequence**: Auto-incrementing number
   - 1 to 4 digits (configurable per typification)
   - Automatically generated by the system
   - Ensures uniqueness
   - Padded with leading zeros (e.g., `001`, `0001`, `00001`)

### Examples

- `SRV-WEB-proj1-001` - Server VM, web server, project 1, first VM
- `PRD-DB-appname-0001` - Production DB, appname, first VM
- `TST-APP-feature1-0001` - Test App, feature1, first VM
- `pre1-prefijo2-manual-001` - Custom prefixes, manual value, first VM

## API Endpoints

### Typification Management

#### Create Typification
```
POST /typing/templates

Request Body:
{
  "name": "tp-haki",
  "prefijo1": "SRV",
  "prefijo2": "WEB",
  "seq_digits": 3
}

Response:
{
  "id": 1,
  "name": "tp-haki",
  "prefijo1": "SRV",
  "prefijo2": "WEB",
  "seq_digits": 3,
  "is_active": true,
  "created_at": "2026-02-02T12:00:00Z"
}
```

**Validation:**
- `name`: Required, 1-100 characters, unique
- `prefijo1`: Required, alphanumeric only, 1-50 characters
- `prefijo2`: Required, alphanumeric only, 1-50 characters
- `seq_digits`: Required, integer 1-4

#### List Typifications
```
GET /typing/templates

Response:
[
  {
    "id": 1,
    "name": "tp-haki",
    "prefijo1": "SRV",
    "prefijo2": "WEB",
    "seq_digits": 3,
    "is_active": true,
    "created_at": "2026-02-02T12:00:00Z"
  }
]
```

**Notes:**
- Only returns active typifications
- Typifications marked as inactive (edited) are not shown

#### Update Typification
```
PUT /typing/templates/{template_id}

Request Body:
{
  "prefijo1": "SRV",
  "prefijo2": "DB",
  "seq_digits": 4,
  "edit_reason": "Updated to match new naming standards (migration to 4 digits)"
}

Response:
{
  "id": 2,
  "name": "tp-haki (v1-1738500000)",
  "prefijo1": "SRV",
  "prefijo2": "DB",
  "seq_digits": 4,
  "is_active": true,
  "created_at": "2026-02-02T12:05:00Z"
}
```

**Behavior:**
- Creates a NEW typification with the updated values
- Marks the ORIGINAL typification as `is_active: false`
- Stores the `edit_reason` in the original typification
- Copies the current sequence counter to the new typification
- The original typification's name is appended with version info

**Validation:**
- `edit_reason`: Required, 10-255 characters
- Other fields follow same rules as creation
- At least one field (prefijo1, prefijo2, or seq_digits) must be provided

### VM Name Generation

#### Preview VM Name
```
POST /typing/generate-name/{template_id}

Request Body: "proj1"

Response:
{
  "full_name": "SRV-WEB-proj1-001",
  "segments": ["SRV", "WEB", "proj1", "001"],
  "next_seq": 1
}
```

**Behavior:**
- Validates that `manual_value` contains only alphanumeric characters
- Generates the next sequence number (but does NOT increment it)
- Returns the preview of the VM name
- Useful for showing the user what the VM name will be

**Validation:**
- `manual_value`: Required, alphanumeric only, non-empty

## User Guide

### For Administrators

#### Creating a New Typification

1. Navigate to the **Typifications** page
2. Click **"New Typification"** button
3. Fill in the form:
   - **Name**: A descriptive name for the typification (e.g., "Production Web Servers")
   - **Prefijo 1**: First prefix (e.g., "SRV")
   - **Prefijo 2**: Second prefix (e.g., "WEB")
   - **Sequence Digits**: Choose 1, 2, 3, or 4 digits
4. Review the **Preview** to see the naming pattern
5. Click **"Create"**

**Example Configuration:**
- Name: `Production Web Servers`
- Prefijo 1: `PRD`
- Prefijo 2: `WEB`
- Sequence Digits: 4
- Result: `PRD-WEB-{MANUAL}-0001`, `PRD-WEB-{MANUAL}-0002`, etc.

#### Editing an Existing Typification

1. Go to the **Typifications** page
2. Find the typification you want to edit
3. Click the **"Edit"** button
4. Modify the fields you want to change
5. Provide a mandatory **Edit Reason** (10+ characters)
6. Review the preview
7. Click **"Update"**

**What Happens:**
- A NEW typification is created with the updated values
- The ORIGINAL typification is marked as inactive
- VMs created with the original typification can still be found (they're not deleted)
- New VMs will use the updated typification

**Why This Approach?**
- Preserves history of typifications
- Allows VMs created with old typifications to remain accessible
- Prevents unexpected changes to existing VM names
- Maintains audit trail through edit reasons

### For VM Creators

#### Creating a VM with a Typification

1. Navigate to the **Dashboard**
2. Fill in the VM creation form:
   - **Typification**: Select a typification from the dropdown
   - **Manual Value**: Enter the manual identifier (e.g., "proj1")
   - **Description**: VM description
   - **Template**: VM template (CPU, RAM, Disk specs)
   - **vCenter Info**: Datacenter, cluster, resource pool
3. Review the **VM Name Preview** to see what the name will be
4. Click **"Create VM"**

**Example:**
- Typification: "Production Web Servers" (PRD-WEB-{MANUAL}-000X)
- Manual Value: "proj1"
- Resulting VM Name: `PRD-WEB-proj1-0001`

**Validation:**
- Manual value must contain only letters and numbers
- VM name preview shows the actual name that will be generated
- The sequence number is automatically incremented when the VM is created

## Orphaned VMs

### What Are Orphaned VMs?

Orphaned VMs are virtual machines created with a typification that has been edited (marked as inactive). These VMs:

- Still exist and are fully functional
- Can still be managed, viewed, and appear in statistics
- Have a special tag or label indicating they use an orphaned typification
- Are distinct from VMs created without this system (not our responsibility)

### Identifying Orphaned VMs

VMs created with inactive typifications will have:

1. A typification ID pointing to a typification with `is_active: false`
2. The typification's `edit_reason` field populated with the reason for deactivation

### Example Scenario

1. **Initial State:**
   - Typification "Standard" created: `SRV-WEB-{MANUAL}-001`
   - VMs created: `SRV-WEB-proj1-001`, `SRV-WEB-proj2-002`

2. **Typification Updated:**
   - "Standard" typification edited to: `SRV-WEB-{MANUAL}-0001` (4 digits)
   - Edit reason: "Migration to 4-digit sequence"
   - Original "Standard" marked as `is_active: false`

3. **Result:**
   - Existing VMs (`SRV-WEB-proj1-001`, `SRV-WEB-proj2-002`) are **ORPHANED**
   - New VMs use updated typification: `SRV-WEB-{MANUAL}-0001`
   - Orphaned VMs can still be managed and are marked as using an orphaned typification

## Technical Details

### Database Schema

#### Typification Template Table
```sql
CREATE TABLE typification_templates (
    id INTEGER PRIMARY KEY,
    name VARCHAR(100) UNIQUE NOT NULL,
    prefijo1 VARCHAR(50) NOT NULL,
    prefijo2 VARCHAR(50) NOT NULL,
    seq_digits INTEGER NOT NULL,
    is_active BOOLEAN DEFAULT TRUE,
    edit_reason VARCHAR(255),
    created_by INTEGER REFERENCES users(id),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

#### Typification Counter Table
```sql
CREATE TABLE typification_counters (
    template_id INTEGER REFERENCES typification_templates(id) PRIMARY KEY,
    current_value INTEGER DEFAULT 0,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

#### VM Provision Table
```sql
CREATE TABLE vm_provisions (
    id INTEGER PRIMARY KEY,
    vm_name VARCHAR(255) UNIQUE NOT NULL,
    template_id INTEGER REFERENCES typification_templates(id),
    requester_id INTEGER REFERENCES users(id),
    vcenter_datacenter VARCHAR(100),
    vcenter_cluster VARCHAR(100),
    vcenter_resource_pool VARCHAR(100),
    status VARCHAR(20) DEFAULT 'pending',
    specs JSON,
    error_log TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

### Sequence Generation Logic

1. When a VM is created:
   - The system queries the counter for the typification
   - Increments the counter by 1
   - Pads the number with leading zeros based on `seq_digits`
   - Generates the VM name: `{prefijo1}-{prefijo2}-{manual_value}-{sequence}`

2. When previewing a VM name:
   - The system calculates what the next number would be
   - Does NOT increment the counter
   - Returns the preview for user confirmation

3. Counter behavior:
   - Starts at 0
   - First VM: 1 (padded to 001, 0001, etc.)
   - Subsequent VMs: 2, 3, 4, etc.
   - Counter is persisted across application restarts
   - When a typification is edited, the counter value is copied to the new typification

## Best Practices

### Naming Conventions

1. **Use meaningful prefixes:**
   - Environment codes: `PRD`, `TST`, `DEV`, `STG`
   - Service types: `SRV`, `WEB`, `DB`, `APP`, `LB`, `GW`
   - Location codes: `BA`, `NY`, `LDN`, `TKY`

2. **Keep manual values consistent:**
   - Project codes: `proj1`, `proj2`, `featureX`
   - Owner codes: `john`, `alice`, `teamA`
   - Purpose codes: `billing`, `reporting`, `api`

3. **Choose appropriate sequence length:**
   - 1 digit: Small deployments (<10 VMs)
   - 2 digits: Medium deployments (<100 VMs)
   - 3 digits: Large deployments (<1000 VMs)
   - 4 digits: Very large deployments (<10000 VMs)

### Typification Management

1. **Plan typifications carefully:**
   - Think about long-term naming strategies
   - Consider different environments (prod, test, dev)
   - Account for different services (web, db, app)
   - Plan for future growth (sequence digit length)

2. **Document typifications:**
   - Keep a record of what each typification is for
   - Note when and why typifications were edited
   - Track which typifications are being used actively

3. **Review orphaned VMs:**
   - Periodically check for VMs using orphaned typifications
   - Determine if they need migration or cleanup
   - Maintain audit trail of changes

## Troubleshooting

### Common Issues

#### Issue: "Typification with these prefixes already exists"
**Solution:** Choose a different combination of prefijo1 and prefijo2, or edit the existing one

#### Issue: "Manual value must contain only letters and numbers"
**Solution:** Remove special characters, spaces, or other non-alphanumeric characters from the manual value

#### Issue: VM name not as expected
**Solution:** 
1. Verify the typification configuration
2. Check that the manual value is correct
3. Ensure the sequence counter hasn't been reset

#### Issue: Cannot find VMs created with old typification
**Solution:** 
1. The VMs still exist and are not deleted
2. They are marked as using an orphaned typification
3. They can be found by searching for the VM name or filtering by typification status

## Migration Guide

### From Old Naming Convention

If migrating from a different naming system:

1. **Analyze existing VMs:**
   - Extract naming patterns
   - Identify common prefixes
   - Document current sequences

2. **Create new typifications:**
   - Map existing prefixes to typifications
   - Set sequence digits to accommodate existing VM count
   - Provide meaningful edit reasons

3. **Create orphaned typifications:**
   - Create typifications matching old naming patterns
   - Mark them as inactive immediately
   - This ensures existing VMs are properly tracked

4. **Update procedures:**
   - Update documentation
   - Train team on new process
   - Update scripts and tools to use new system
